# 工厂方法模式

# 简介

工厂方法模式的实质是，通过让工厂子类决定实例化那个类，Factory Method 使一个类的实例化延迟到了其子类。

实现了创建者与调用者的分离，调用者无需关注其创建过程，只需选择自己想要使用的即可。简单通俗来说，就像一个大型的工厂，对于消费者来说，只需要知道都有什么工厂的产品生产出来，而不需要关心工厂是如何生产产品的。对于工厂来说，必须知道所有的产品的制造方法。

# 代码示例一

实现一个导出数据的应用框架,来让客户来选择数据的导出方式（txt格式、excel格式、xml格式）,并真正执行数据导出。

# 使用工厂方法模式

  1. 实现导出文件接口ExportFileApi
  2. 实现文件接口ExportFileApi的实现类
public interface ExportFileApi{
  public boolean export(String data);
}

public class ExportTxtFile implements ExportFileApi{
  public boolean export(String data){
    Sysetm.out.println("TxtFileExport.export"+data);
    return true;
  }
}
public class ExportExcelFile implements ExportFileApi{
  public boolean export(String data){
    Sysetm.out.println("ExcelFileExport.export"+data);
    return true;
  }
}
public class ExportXmlFile implements ExportFileApi{
  public boolean export(String data){
    Sysetm.out.println("XmlFileExport.export"+data);
    return true;
  }
}
  1. 实现ExportOperate
public abstract class  ExportOperate{
  public boolean export(String data){
    ExportFileApi api = new factoryMethod();
    return api.export(data);
  }

  protected abstract ExportFileApi factoryMethod();
}

public class ExportTxtFileOperate extends  ExportOperate{
  protected ExportFileApi factoryMethod(){
    return new ExportTxtFile();
  }

}
public class ExportExcelFileOperate implements ExportOperate{
  protected ExportFileApi factoryMethod(){
    return new ExportExcelFile();
  }
}
public class ExportXmlFile implements ExportFileApi{
  protected ExportFileApi factoryMethod(){
    return new ExportXmlFile();
  }
}
  1. 客户端使用需要使用的create对象，然后调用相应的功能。
public class  Client{
  public static void main(String[] args){
    ExportOperate exportOperate = new ExportTxtFileOperate();
    exportOperate.export("测试数据");
  }
}

# 工厂模式优缺点

工厂模式优点

  • 可以在不知具体实现的情况下编程
  • 工厂方法模式可以让你在实现功能的时候，如果需要某个产品对象，只需要使用产品接口即可，而无需关心具体的实现。选择具体实现的任务延迟到子类去完成。

工厂模式的缺点

  • 具体产品对象和工厂方法的耦合性。
  • 在工厂方法模式中，工厂方法是需要创建产品对象的，也就是需要选择具体的产品对象，并创建他们的实例，因此具体产品对象和工厂方法是耦合的。

# 适用场景

  • 如果一个类需要创建某个接口的对象，但是又不知道具体的实现，这种情况可以选用工厂方法模式，把创建对象的工作延迟到子类中去实现。
  • 如果一个类本身就希望由它的子类来创建所需的对象的时候，应该使用工厂方法模式。

# 相关设计模式

# 依赖倒置原则

依赖倒置要求“要依赖抽象，而不是依赖具体实现”，不能让高层组件依赖于底层组件，而不管高层组件还是底层组件，都应该依赖于抽象。

比如前面的示例，实现客户端请求操作的ExportOperate 就是高层组件，而具体实现数据导出的对象就是底层组件，比如 ExportTxtFile、ExportExcelFile、ExportXmlFile，而ExportFileApi接口就相当于那个抽象。

对于 ExportOperate 来说，它不关心具体的实现方式，它只是“面向接口编程”，对于具体的实现来说，它只关心自己“如何实现接口”所要求的功能。

其中倒置的就是这个接口的“所有权”，事实上，ExportFileApi接口中定义的功能，都是高层组件提出的要求，也就是说接口中的功能是高层组件需要的功能，但是高层组件只是提出要求，并不关心如何实现，而底层组件就是来真正实现高层组件所要求的接口功能的，因此看起来，底层实现的接口的所有权并不在底层组件手中，而是倒置到高层组件去了。

# 工厂方法模式和模板方法模式

这两个模式外观相似，都有一个抽象类，然后由子类来提供一些实现，但是工厂方法模式的子类专注的是创建产品对象，而模板方法模式的子类专注的是为固定的算法骨架提供某些步骤的实现。
这两个模式可以组合使用，通常在模板方法模式里面，使用工厂方法来创建模板方法需要的对象。
